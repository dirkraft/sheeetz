# Sheeetz

Sheet music library manager with PDF metadata editing, Google Drive integration, and a browser-based viewer.

## Architecture

```
backend/          FastAPI + SQLAlchemy (async SQLite)
frontend/         Vue 3 + TypeScript + Vite
e2e/              Playwright end-to-end tests
```

### Backend

**FastAPI** app serving four router modules under the root path (Vite proxies `/api` → backend):

| Router | Prefix | Purpose |
|--------|--------|---------|
| `auth` | `/auth` | Google OAuth 2.0 login, session cookies, user settings |
| `sheets` | `/sheets` | List/get/download sheets, metadata CRUD, metadata key/value autocomplete |
| `folders` | `/folders` | Browse local/Drive folders, add library folders, trigger scans |
| `admin` | `/admin` | Clear index (bulk delete all sheets for current user) |

**Database:** SQLite via `aiosqlite`, managed with Alembic migrations. Four tables:

- `users` — Google identity, Drive tokens, settings JSON
- `sheets` — indexed PDFs (backend-agnostic: `backend_type` + `backend_file_id`)
- `sheet_meta` — EAV table for all metadata (key/value pairs per sheet)
- `library_folders` — user-selected folders to scan

**Auth:** Google OAuth 2.0 with `itsdangerous` session cookies (7-day TTL). The `get_current_user` dependency gates all non-auth routes.

### Frontend

**Vue 3** (Composition API, `<script setup>`) with Vue Router. No global state library — each view manages its own state with `ref`/`computed`.

| Route | View | Purpose |
|-------|------|---------|
| `/` | Home | Login/logout |
| `/library` | Library | Manage library folders, trigger scans |
| `/sheets` | Sheets | Paginated table with filters, sortable/configurable columns |
| `/sheets/:id` | SheetViewer | PDF viewer (pdfjs-dist) + metadata edit panel |
| `/admin` | Admin | Clear index |

**API layer:** Single `apiFetch<T>` wrapper in `api.ts` handles all backend calls with `credentials: 'include'`.

## Key Design Decisions

### PDF Metadata (non-destructive editing)

Metadata is read from both PDF DocInfo and XMP during scan. User edits are written only to a custom XMP namespace (`http://sheeetz.app/meta/1.0/`), leaving original `dc:`, DocInfo, etc. untouched. On re-scan, the sheeetz namespace takes priority (first-match-wins in `RAW_TO_CORE_MAP`).

`pages` is always auto-derived from `len(pdf.pages)` and cannot be overwritten.

### EAV Metadata Storage

`sheet_meta` is a classic Entity-Attribute-Value table: `(sheet_id, key, value)`. This allows arbitrary metadata fields without schema changes. The list endpoint supports:
- Multi-key filtering via aliased JOINs (AND semantics)
- Sorting on any metadata key via LEFT OUTER JOIN

### Dynamic Columns

The Sheets list has two types of columns:
- **Static columns** — hardcoded core fields (filename, title, composer, tags, pages, folder, source)
- **Custom columns** — discovered dynamically from `GET /sheets/metadata/keys` on mount and from sheet metadata after each load

Custom columns appear in the column picker with a "custom" label. Column selection and order are persisted to user settings.

### Backend-Agnostic Storage

`Sheet.backend_type` is `"local"` or `"gdrive"`. `Sheet.backend_file_id` is an absolute filesystem path or a Google Drive file ID. The unique constraint `(user_id, backend_type, backend_file_id)` prevents duplicates. Router code switches on `backend_type` for download/upload operations.

## Development

### Prerequisites

- Python 3.12+
- Node.js 20+
- tmux
- Google OAuth credentials (for Drive integration; local-only mode works without)

### Initial Setup

```bash
cd backend
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt

# Configure (or copy .env.example)
cat > .env <<EOF
SECRET_KEY=your-secret
GOOGLE_CLIENT_ID=...
GOOGLE_CLIENT_SECRET=...
GOOGLE_REDIRECT_URI=http://localhost:8000/auth/callback
FRONTEND_URL=http://localhost:5173
ENABLE_LOCAL_BACKEND=true
EOF
```

```bash
cd frontend
npm install
```

### Local Stack (Preferred)

```bash
cd backend
source .venv/bin/activate
sheeetz up
```

The `sheeetz` CLI in `backend/sheeetz/cli.py` is the source of truth for local stack management. Use it for stack lifecycle commands instead of manually starting backend/frontend in separate shells.

If needed, direct app entry points are:
- backend app: `backend/sheeetz/main.py`
- frontend dev server: `frontend/package.json` (`npm run dev`)

### Database Migrations

```bash
cd backend
alembic upgrade head        # apply all migrations
alembic revision --autogenerate -m "description"  # create new migration
```

## Testing

### API Tests

```bash
cd backend
pytest tests/ -v --ignore=tests/test_drive.py
```

Uses in-memory SQLite. Drive API calls are mocked with `respx` — no Google credentials needed.

### E2E Tests

```bash
cd e2e
npm install
npx playwright install chromium --with-deps
npx playwright test
```

Playwright spins up both backend (port 8001) and frontend (port 5174) automatically. Test fixtures are real PDFs generated by `backend/tests/generate_fixtures.py` with pikepdf.
Use the `sheeetz` stack CLI for interactive local development; E2E does not require an already-running local stack.

### CI

GitHub Actions runs on PRs to `main`:
- **api-tests** — Python 3.12, pytest (excluding Drive tests unless secrets are configured)
- **e2e-tests** — Full Playwright suite with Chromium

## Project Structure

```
backend/
  sheeetz/
    main.py              # FastAPI app, CORS, router registration
    config.py            # pydantic-settings (env vars / .env)
    auth.py              # Google OAuth helpers, token signing
    db.py                # async SQLAlchemy engine + session
    models.py            # User, Sheet, SheetMeta, LibraryFolder
    routers/
      auth.py            # /auth/* — login, callback, me, settings
      sheets.py          # /sheets/* — list, detail, PDF, metadata
      folders.py         # /folders/* — browse, add, scan
      admin.py           # /admin/clear-index
    storage/
      metadata.py        # PDF metadata extract/map/write (pikepdf)
      scanner.py         # scan_local_folder, scan_gdrive_folder
      drive_api.py       # Google Drive v3 REST wrappers (httpx)
      local.py           # Local filesystem helpers
  tests/
    generate_fixtures.py # Creates test PDFs with XMP metadata
  alembic/               # Database migrations

frontend/
  src/
    api.ts               # All API calls + TypeScript interfaces
    router.ts            # Vue Router config
    views/
      Sheets.vue         # Table + filters + dynamic column picker
      SheetViewer.vue    # PDF viewer + metadata edit panel
      Library.vue        # Folder management + scan
    components/
      AutocompleteInput.vue  # Typeahead input
      FolderPicker.vue       # Folder browse modal

e2e/
  seed.py                # SQLite seed + token generation for tests
  tests/
    global-setup.ts      # Generate fixtures, seed DB, inject auth
    columns.spec.ts      # Column picker behavior
    sheets.spec.ts       # Sheet list + filtering
    viewer.spec.ts       # PDF viewer + metadata editing
    library.spec.ts      # Folder scan
    admin.spec.ts        # Clear index
```
